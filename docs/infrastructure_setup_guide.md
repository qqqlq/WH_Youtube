# AVAP インフラ構築ガイド（初心者向け解説）

このドキュメントは、AVAPプロジェクトを外部に公開するために組んだ「インフラ（システム基盤）の仕組み」を、**駆け出しエンジニアでも理解できる**ようにやさしく解説したものです。

---

## 🏗️ 全体像：どうやって外部からアクセスしているの？

AVAPのシステムは、構成を大きく2つに分けています：
1. **フロントエンド（画面）**: 今後 Vercel (バーセル) というクラウドサービスに置きます。
2. **バックエンド（API・動画生成部）**: あなたの自宅の「ミニPC」の上で動いています。

問題は、**「Vercel（外の世界）から、どうやって安全に自宅のミニPC（内の世界）と通信させるか？」** です。ここで活躍するのが **systemd** と **Cloudflare Tunnel** という2つの技術です。

---

## 🔌 登場人物1： systemd (システムディー)

### 何をしてくれるもの？
「あなたが寝ていても、パソコンが再起動しても、**指定したプログラムを裏側でずっと見張って動かし続けてくれる**」Linuxの標準機能です。

### なぜ必要なの？
これまでFastAPIサーバーを動かすときは、ターミナルで `uvicorn server.app:app` というコマンドを打っていましたよね。でも、これには大きな弱点があります。
- ターミナルを閉じるとサーバーも止まってしまう
- エラーで落ちたら、誰かが手動で叩き直すまでそのまま
- パソコンを再起動したら、また自分でコマンドを打たないといけない

### 今回はどう設定した？
`~/.config/systemd/user/avap.service` という設定ファイルを作りました。
これに「`uvicorn`でサーバーを立ち上げてね」「もし落ちたら5秒後に自動で再起動してね(`Restart=always`)」と書き込みました。
これにより、AVAPのAPIサーバーは**「絶対に死なない不死身のプロセス」**として、常にポート `8000` で待機するようになりました。

---

## 🚇 登場人物2： Cloudflare Tunnel (クラウドフレア トンネル)

### 何をしてくれるもの？
「ルーターのポートを一切開けずに、自宅のPCと世界のインターネットを安全な**『魔法のトンネル』**で繋いでくれる」サービスです。今回は `avap-api.nodewalker.app` という独自ドメインと紐付けています。

### なぜ必要なの？
昔は、自宅のサーバーを公開するなら「ルーターの8000番ポートに穴を開けて通す（ポート開放）」のが普通でした。しかしこれは、**家の玄関の鍵を開けっぱなしにするのと同じ**くらい危険です。世界中からハッカーがやってきてサイバー攻撃を受けます。

### 仕組みはどうなっているの？
1. ミニPCにインストールした `cloudflared` というプログラムが、**「内側から外側（Cloudflareの最強サーバー群）へ」** 向かって専用のトンネルを彫ります。
2. Cloudflareのサーバーが `https://avap-api.nodewalker.app` というURLで門番になり、外からのアクセスを受け取ります（ここで不審者は全て弾かれます）。
3. 安全なお客さん（VercelなどからのAPIリクエスト）だけが、そのトンネルを通って、ミニPCの `8000` 番ポートに直接届けられます。

### すごいところ
- **ルーターの穴あけ不要**: 内側から外に出る通信なので、ハッカーからはあなたの家のIPアドレスが全く見えません。
- **HTTPS（暗号化）が全自動**: これまでNginxなどを使って手動でやっていたSSL証明書の設定を、Cloudflareが勝手にやってくれます。
- **DDoS防御**: もし1秒間に100万回のアクセス攻撃を受けても、すべてCloudflareの超巨大サーバーが盾になって防いでくれます。ミニPCは涼しい顔でいられます。

## 📝 今後の運用のためのコマンド集

システムがうまく動かなくなった時は、以下のコマンドで状態をチェックしたり再起動したりできます。

### サーバー本体 (FastAPI) の管理
```bash
# 状態を見る（エラーログの確認に便利）
systemctl --user status avap.service

# 設定を変えた時などに再起動する
systemctl --user restart avap.service

# サービスを止める
systemctl --user stop avap.service
```

## 【補足】Vercelでカスタムドメイン（サブドメイン）を公開する仕組み

今後、別のアプリを作って `app2.nodewalker.app` などの新しいサブドメインで Vercel にデプロイしたい場合も、**今回と同じようにCloudflareでレコードを追加する作業が必要**になります。

### 用語の解説

1. **CNAME (Canonical Name)**
   - 「このサブドメインのアクセス先は、あっちのドメイン名と同じだよ」と**転送（別名定義）**するための設定です。
   - VercelなどのクラウドサービスはサーバーのIPアドレスがころころ変わるため、IPアドレス(Aレコード)ではなく、常にVercelへ向かう固定のドメイン名（`cname.vercel-dns.com`）を指定してアクセスを流します。

2. **Target (ターゲット / コンテンツ)**
   - 転送先の「あっちのドメイン名」のことです。
   - 今回の場合は Vercel のサーバー群を指す `cname.vercel-dns.com` がターゲットになります。

3. **Proxy status (プロキシステータス / オレンジの雲)**
   - **オレンジの雲 (Proxied)**: アクセス者があなたのサーバー(Vercel)に到達する前に、**Cloudflareの強力な壁（CDN・ファイアウォール）を一度経由**させます。これによってセキュリティが上がり、表示が高速化されます。
   - **グレーの雲 (DNS Only)**: Cloudflareの壁を通さず、単なる「道案内（DNS）」として扱います。

### なぜ Vercel では「グレーの雲 (DNS Only)」にしなければならないのか？
Vercel 自体が**すでにCloudflareと同じ、世界最高峰のエッジネットワーク（独自の強力な壁とSSL証明書）を持っているから**です。

Cloudflareのオレンジの雲をオンにしたままVercelに繋ごうとすると、**「壁が2重」**になってしまい、通信経路がぶつかって「Proxy Detected」エラーになったり、暗号化（SSL）のやり取りが上手くいかず画面が表示されなくなってしまいます。

そのため、Vercel 等の優秀なホスティングサービスを使う場合は、Cloudflare側は「グレーの雲」にして単なる道案内に徹してもらうのが正解です。
```bash
sudo systemctl status cloudflared

# トンネルを再起動する
sudo systemctl restart cloudflared
```
